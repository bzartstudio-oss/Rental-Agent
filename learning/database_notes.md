# Database Notes

Storage/database-specific lessons and decisions for this project. See [Project Learning.md](Project%20Learning.md) for the full index.

## Decisions

- **2026-07-14 — Storage engine: SQLite**, single file at `data/rental_intelligence.db`. Chosen over flat JSON files because the "reproducible and comparable over time" and versioning principles ([../docs/00_Project_Vision.md](../docs/00_Project_Vision.md)) need real relational queries — price trend for one apartment, diffing two search runs — which flat files make painful. Single-machine/single-user in V1, so no server/concurrency requirements pushed toward anything heavier. Zero extra dependency (`sqlite3` is Python standard library). Full schema: [../docs/03_Data_Model.md](../docs/03_Data_Model.md).
- **2026-07-14 — `search_results` is an immutable snapshot table**, not just a join to live `apartments` data. It denormalizes `price_at_search`/`status_at_search` at write time. Without this, an old report's numbers would silently change as new searches update `apartments.current_price` — which breaks "every search is reproducible and comparable over time" the moment you re-read an old report. See [../docs/03_Data_Model.md](../docs/03_Data_Model.md).
- **2026-07-14 — `data/apartments/`, `data/search_history/`, `data/platform_registry/` (scaffolded 2026-07-13) are superseded by SQLite tables.** They're not deleted, just no longer where new code should write — see [../docs/02_Folder_Guide.md](../docs/02_Folder_Guide.md).
- **2026-07-14 — No `migrations/` folder built in Phase 1**, despite one being sketched in the original [../docs/02_Folder_Guide.md](../docs/02_Folder_Guide.md) package tree. `storage/database.py` applies `schema.sql` directly via `CREATE TABLE IF NOT EXISTS` on every connection, which is idempotent and sufficient until the schema actually needs to change under existing data. A migrations framework before there's ever been a second migration would be exactly the kind of premature abstraction [CLAUDE.md](../CLAUDE.md) warns against — add one when the first real schema change is needed, not before.
- **2026-07-14 — That moment arrived: the Migration Framework (Sprint V2.0.1).** `storage/migrations/` + a `schema_migrations` table now exist. `storage/database.py` applies `schema.sql`, then any not-yet-applied migration file, sorted by **numeric** version (not filesystem order), each in its own real transaction. `0001_v2_knowledge_engine.sql` carries the entire Version 2.0 schema — 6 new tables, ~18 new nullable columns across 4 existing tables, all additive. Verified against the real dev database (which had genuine v1.1 data): migrated in place, zero data loss, no reset. Full design rationale: [../docs/10_Roadmap.md](../docs/10_Roadmap.md) "Migration Plan".

## Lessons Learned

- **2026-07-14 — Python's `sqlite3` module does not implicitly open a transaction before DDL statements (`CREATE TABLE`, `ALTER TABLE`) the way it does for DML (`INSERT`/`UPDATE`/`DELETE`).** Building the migration runner's rollback path, a `CREATE TABLE` executed via `conn.execute()` inside the existing `Database.transaction()` helper (which relies on `commit()`/`rollback()` around Python's implicit-transaction heuristic) committed immediately — a later `rollback()` after a subsequent statement failed did nothing to undo it. SQLite itself fully supports transactional DDL; getting Python's driver to honor that requires bypassing the implicit heuristic entirely with an explicit `BEGIN`/`COMMIT`/`ROLLBACK` on a connection with `isolation_level = None`. Fixed in `storage/database.py::_apply_one_migration` — `Database.transaction()` itself is untouched and remains correct for its actual use (every repository does DML only). See [python_notes.md](python_notes.md) for the full writeup. Lesson: a migration/DDL system's "rollback" claim is unverified until there's a test with a migration that's *designed* to fail partway through — the happy-path tests gave zero signal that rollback was broken.
- **2026-07-14 — A row's `search_id` is a more reliable "which run wrote this" signal than its `observed_at` timestamp compared against that run's `created_at`.** Building Search Memory's run-over-run comparison (v2.0 Step 3), a design based on `previous.created_at < observed_at <= current.created_at` broke against the real pipeline: `SearchRequest.created_at` is stamped before any processing happens, so a search's *own* writes (always later, since processing takes real time) satisfy `observed_at > that_same_search.created_at` — meaning a run's own initial data looked like "a change relative to itself" to the very next comparison. Fixed by matching `entry.search_id == search.id` first, falling back to the timestamp only for rows genuinely outside the two searches being compared. Lesson: when reconstructing "the state as of event E" from a table that also has an explicit foreign key back to E, prefer matching on that key over a timestamp comparison against E's own recorded time — the timestamp only tells you *when* something happened, not *which logical step* it belongs to, and those can diverge by however long processing takes. Full writeup in [architecture_notes.md](architecture_notes.md) and [../docs/17_Search_Memory.md](../docs/17_Search_Memory.md) "A Real Bug".
