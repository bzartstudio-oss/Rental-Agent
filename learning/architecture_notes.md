# Architecture Notes

Architecture decisions and project-structure lessons for this project. See [Project Learning.md](Project%20Learning.md) for the full index.

## Decisions

- **Canonical project location is `Clients/Rental-Agent`.** An earlier duplicate scaffold at `Clients/Agent Rental/Rental-Agent` was created by mistake and has been removed — its content had already been moved into the real folder before that was noticed.
- **2026-07-13 — Rental type decided: residential apartments.** Confirmed when creating the `data/apartments/` subfolder made the implicit choice visible and it was checked against the user rather than assumed. Vacation/short-term and equipment rentals are out of scope unless revisited. This closed the open question in [../notes/Questions.md](../notes/Questions.md) and updated `Listing` fields in [../docs/03_Data_Model.md](../docs/03_Data_Model.md) to residential-specific ones (bedrooms, bathrooms, sqft).
- **2026-07-13 — `data/` subfolder structure defined**: `platform_registry/`, `apartments/`, `knowledge_base/`, `search_history/`, `media/`, `raw_pages/`, `cache/` — see [../docs/02_Folder_Guide.md](../docs/02_Folder_Guide.md) for what each holds. Whether the generated/large ones (`cache/`, `raw_pages/`, `media/`, and possibly `apartments/`/`search_history/`) should be gitignored is explicitly deferred until there's real data to gauge size against — tracked in [../notes/Questions.md](../notes/Questions.md), don't assume an answer before then.
- **2026-07-13 — Established a documentation-first workflow via a numbered `docs/` structure (`00_Project_Vision.md` through `14_Lessons_Learned.md`).** Rationale: several core decisions (target rental type, data source, ranking criteria) weren't made yet, so the docs were written as skeletons with explicit `*TBD*` markers rather than invented content — this gives every future implementation piece a defined place to live and a place to record its "why" without pretending unmade decisions are settled. See [../docs/13_Claude_Guidelines.md](../docs/13_Claude_Guidelines.md) for the convention going forward, and [../docs/11_Development_Journal.md](../docs/11_Development_Journal.md) for a chronological log distinct from this file's curated one.
- **2026-07-13 — Created the target `src/` module structure**: `core/`, `connectors/`, `discovery/`, `search/`, `collectors/`, `analyzers/`, `ranking/`, `services/`, `storage/`, `utils/`, `ui/`. `core/` and `utils/` already existed and were left as-is; `analyzer/` (singular, pre-existing, empty) was renamed to `analyzers/` to match.
- **2026-07-13 — `learning/` split into topic files** (this file and its siblings) instead of one growing `Project Learning.md`. `Project Learning.md` is now an index plus catch-all — see [CLAUDE.md](../CLAUDE.md) for the routing convention.

## Lessons Learned / Known Issues

- **Known issue, deliberately not fixed yet: `src/` has pre-existing folders that don't map onto the structure above** — `ai/`, `browser/` (has `browser_manager.py`), `config/` (has `config_loader.py`), `data_source/`, `exporters/`, `filters/`, `maps/`, `model/` (has `apartment.py`), `models/` (has `configuration.py`), `reports/`. `model/` vs `models/` in particular looks like an accidental duplicate. This code predates the docs-first architecture in [../docs/01_System_Architecture.md](../docs/01_System_Architecture.md) and doesn't follow its naming. Explicitly deferred — reconcile (map old code into `connectors/`, `storage/`, etc., or delete duplicates) before or while writing new code in the overlapping areas, so logic doesn't end up split across both an old and a new folder for the same concern.

- **2026-07-13 — Incident: unexplained batch file wipe across the project.** Within roughly a one-minute window, ~40 files across `docs/`, `notes/`, `data/`, `src/`, and the project root were force-created or truncated to zero bytes — this destroyed the content of all 15 `docs/00`–`14` files (which had real content moments before), and left behind a pile of empty, unexplained files that didn't match this project's conventions: `docs/decisions/001_Decision.md` through `020_Decision.md` (plus oddly specific ones — `001_SearchRequest.md`, `002_PlatformRegistry.md`, `003_DiscoveryAgent.md`, `004_RankingSystem.md` — that echo this project's own pipeline terminology), `CHANGELOG.md`, `PROJECT_RULES.md`, `TODO.md`, `examples/README.md`, `scripts/README.md`, `docs/README.md`, `notes/README.md`, `data/README.md`, and empty `__init__.py` files under `src/` and `tests/`. Also found (older, from 2026-07-12, likely from an earlier untracked work session, not this incident): `docs/projectblueprint.md`, `docs/developmentroadmap.md`, `docs/developmentjournal.md` — all empty, functionally superseded by `docs/00_Project_Vision.md`, `docs/10_Roadmap.md`, `docs/11_Development_Journal.md`.
  - **Investigated and ruled out:** no scheduled task (`list_scheduled_tasks`) or cron job (`CronList`) was configured against this project; Desktop is not OneDrive-redirected (checked `HKCU\...\User Shell Folders` and reparse-point status) so it isn't a OneDrive sync conflict.
  - **Root cause: undetermined.** The clustering of all changes within about a minute, uniformly empty content, and a mix of generic (`001_Decision.md`) and oddly-specific-to-this-project filenames is most consistent with a scripted "ensure these files exist" scaffold tool (e.g. something using `New-Item -Force`-style semantics) run once against this folder — not a live/ongoing process. The user confirmed no other session should have been running. **Recommend checking (not yet checked):** other open Claude Code / AI-tool windows or browser tabs pointed at this same folder, any recently-run IDE "generate project" extension, and Windows' own Task Scheduler (distinct from the `scheduled-tasks`/`CronList` tools checked above, which only cover tasks created through this harness).
  - All 15 `docs/00`–`14` files were restored verbatim from the active session's conversation history (the content had been written moments earlier in the same conversation). The unexplained empty leftovers were deleted; `src/browser/__init__.py` (pre-existing before this incident, just now also empty) was left alone.
  - **Fix going forward:** commit meaningful work promptly — see [git_notes.md](git_notes.md) 2026-07-13 entry. Uncommitted work has no recovery path independent of the current conversation.
  - **2026-07-14 addendum:** more casualties from the same incident surfaced later — empty `README.md` files had also been added to `output/` and every `src/` module folder (`analyzers/`, `collectors/`, `core/`, `discovery/`, `ranking/`, `reports/`, `services/`, `storage/`, `ui/`), plus `src/connectors/README.md` (which had real content, written 2026-07-13) was wiped like the `docs/` files. `src/connectors/README.md` was restored from conversation history and updated for the V1.0 design below; the other empty, unexplained `README.md` files were deleted. The empty `__init__.py` files from the same batch were left in place — harmless, and actually consistent with the real Python package structure being designed anyway.

- **2026-07-14 — V1.0 architecture design session: "Rental Intelligence Platform."** The project was reframed from a simple scraper to a self-improving research platform, per 7 core principles now recorded in [../docs/00_Project_Vision.md](../docs/00_Project_Vision.md) (never lose information, every search updates permanent databases, historical versions, reproducible/comparable searches, configurable/extensible filters, extensible to new platforms/cities without redesign, business logic independent of any website). Every numbered doc from `00` through `10` was rewritten to a concrete V1.0 design (not just skeletons) in one session. Key decisions:
  - **Storage engine: SQLite** — see [database_notes.md](database_notes.md) for the full reasoning. This supersedes the plain-file `data/apartments/`, `data/search_history/`, `data/platform_registry/` folders decided 2026-07-13 (not deleted, just no longer the write target — see [../docs/02_Folder_Guide.md](../docs/02_Folder_Guide.md)).
  - **Connectors vs. Collectors split** introduced: `connectors/` holds only platform-specific query/parsing logic; `collectors/` holds generic, reusable fetch infrastructure (browser, HTTP, image download, raw-page persistence) that connectors call into. This is what the Principle-7 "no website-specific logic outside connectors" boundary actually rests on — without it, browser/fetch mechanics would keep getting duplicated per connector. See [../docs/06_Connector_Framework.md](../docs/06_Connector_Framework.md).
  - **Immutable snapshot design for `search_results`**: denormalizes price/status at search time rather than joining to live data, so old reports don't silently change as new searches update `apartments`. See [database_notes.md](database_notes.md).
  - **Legacy `src/` folder reconciliation finally resolved** (deferred since 2026-07-13, see above): concrete disposition for every legacy folder (`browser/` → `collectors/browser_collector.py`, `config/` → `core/config.py`, `model/`/`models/` → `storage/` + delete the duplicate, `data_source/`/`filters/`/`exporters/`/`reports/`/`maps/` → delete, `ai/` → keep, reserved for V2). Table lives in [../docs/02_Folder_Guide.md](../docs/02_Folder_Guide.md); execution is Roadmap Phase 1, not done yet.
  - **Cross-platform apartment de-duplication deliberately deferred to V2** — the schema reserves a `merged_into_id` column now so V2 doesn't need a migration, but V1 only de-duplicates within a single platform. See [../docs/07_Analysis_Engine.md](../docs/07_Analysis_Engine.md).
  - **Extensibility is deliberately asymmetric**: new platforms are a first-class, zero-redesign case (the whole point of the Connector contract); new rental types are *not* pre-built into a generic schema, only documented as an upgrade path — building that generalization now, before a second rental type is ever requested, would violate the project's own "don't design for hypothetical requirements" rule. See "Extensibility Without Over-Engineering" in [../docs/01_System_Architecture.md](../docs/01_System_Architecture.md).
  - Full V1.0 build order, phased with explicit exit criteria per phase, in [../docs/10_Roadmap.md](../docs/10_Roadmap.md) — starts with the storage layer (Phase 1) since every other module depends on it, ends with a second connector (Phase 7) specifically to validate the platform-independence boundary holds under real pressure, not just on paper.
- **2026-07-14 — Phases 1 & 2 implemented and tested** (12 passing tests). Storage foundation (schema, repositories) and Platform Registry + Discovery Agent, both with exit-criteria tests proving the design actually works rather than just existing on paper — see [../docs/10_Roadmap.md](../docs/10_Roadmap.md) for what changed from the original plan (legacy files deleted rather than migrated once confirmed stale). No real platform seeded yet — that's still gated on the open "which platform first" question in [../notes/Questions.md](../notes/Questions.md).
  - **`config/settings.json` and the legacy `src/model/apartment.py` / `src/models/configuration.py` were checked and confirmed stale/experimental** (2026-07-14) — they model a narrower, different concept (room/studio flatshare search in Valencia, specific platforms idealista/fotocasa/badi/spotahome, HTML+PDF reports, walk-distance-to-address filtering) that does not reflect the confirmed V1.0 design. Not used as a basis for the schema. If flatshare-style search (shared rooms, private vs. shared bathroom, distance-to-destination) or PDF reports are wanted later, that's a scope change to raise explicitly, not something to silently reintroduce from this old code.

- **2026-07-14 — Phases 3 through 7 implemented and tested; all V1.0 architecture phases now complete (56 tests passing).** Full detail in [../docs/10_Roadmap.md](../docs/10_Roadmap.md). Key points:
  - **No real rental platform had been picked**, so Phases 3–7 were proven against `demo_platform`/`demo_platform_two` — real `Connector` implementations doing a real Playwright fetch and real BeautifulSoup parse of local HTML fixtures, not live commercial sites. This wasn't a shortcut around the exit criteria (every fetch/parse/DB-write/report-generation is real); only the source is controlled. See [../docs/10_Roadmap.md](../docs/10_Roadmap.md) "Reference Connector Strategy" for the full reasoning — chosen over unilaterally picking a real commercial site to scrape without confirming its ToS.
  - **`analyzers/engine.py` added**, not in the original file list — needed to compose normalizer/deduplicator/change_detector/image-collection into the write sequence from [../docs/07_Analysis_Engine.md](../docs/07_Analysis_Engine.md); that composition can't live in `core/agent.py` (must not contain per-listing business logic) or in any single one of the four analyzer sub-modules alone.
  - **Report generator uses plain Python string templating, not Jinja2** — it isn't an installed project dependency and V1's report layout doesn't need one. Resolves the "proposal, not yet locked in" note that was in [../docs/09_Report_System.md](../docs/09_Report_System.md).
  - **`src/rental_agent.py` retired its old content** (a standalone `OPENAI_API_KEY` status check, from before this architecture existed) and now delegates to `ui/cli.py` as originally planned in [../docs/02_Folder_Guide.md](../docs/02_Folder_Guide.md). Its old test (`tests/test_rental_agent.py`) was deleted along with it — it tested a function that no longer exists.
  - **Test-isolation bug found and fixed**: tests exercising the real `demo_platform` connector were silently writing into real project `data/raw_pages/` and `data/media/` because `raw_page_store.save_page` wasn't being redirected the way `image_collector.collect_image` was. Fixed with a shared `tests/support.py::isolated_collectors()` helper. Full detail in [python_notes.md](python_notes.md). Real project data (`data/raw_pages/demo_platform/`, `data/media/*`, `data/rental_intelligence.db`) was cleaned of test pollution and then repopulated by exactly one deliberate real CLI run (`python -m src.rental_agent --location "Example City" --max-price 2000`), which also surfaced and cleaned up more empty pre-architecture leftovers (`output/exports/`, `output/html/`, `output/pdf/` — matching the deleted `settings.json`'s PDF/export plans, not part of V1.0 scope).
  - **Phase 7's second connector (`demo_platform_two.py`) deliberately uses a different HTML shape** (table/tr/td, different class names, `data-id` vs `data-listing-id`) specifically so its parsing couldn't be copy-pasted from the first connector — a real test of whether the Connector abstraction holds, not just a formality.
  - What's left is explicitly not architectural: pick a real first platform (still open in [../notes/Questions.md](../notes/Questions.md)) and write one more connector for it.

- **2026-07-14 — Version 1.1: Multi-Platform Discovery Framework (73 tests passing, up from 56).** User instruction: build platform discovery/management, explicitly not a real connector yet. Full design in [../docs/05_Platform_Discovery.md](../docs/05_Platform_Discovery.md). Key points:
  - **`platforms` schema reworked**: dropped `base_url`/`connector_module`/`is_active`, added `country`, `supported_cities`, `rental_types`, `homepage`, `search_url`, `requires_login`, `connector_available`, `connector_name`, `last_verified`, `discovery_method`. `is_active` was judged redundant once `connector_available` existed — a platform with no connector can't be searched regardless of a separate active flag, so keeping both would have been two booleans doing one job.
  - **`DiscoveryAgent` split into two roles**: `discover(request)` (search-facing, filters to `connector_available=True`, called by `core/agent.py`) and `sync_platforms(candidates)` (management-facing, implements the 5 required behaviors: load/detect-duplicates/update/save-new/mark-unsupported). Duplicate detection is deliberately simple (exact id or normalized homepage domain only, no fuzzy matching) — see "Why Not Fuzzy Matching" in the doc; nothing has been observed yet that the simple rule misses.
  - **Real-world seed data, no live scraping**: `discovery/known_platforms.py` catalogues 6 real platforms (Zillow, Apartments.com, Rightmove, Idealista, Fotocasa, ImmoScout24 — 4 countries) as `connector_available=False`, alongside the 2 reference connectors as `connector_available=True`. Their names/homepages are public facts; compiling the list made zero live requests to any of them. Fields that would need a live check to verify (`requires_login`, precise `supported_cities`) are left conservative/caveated in `notes` rather than guessed at — matches [CLAUDE.md](../CLAUDE.md)'s "never invent/pretend" principle applied to data, not just code.
  - **Schema change required deleting the dev `data/rental_intelligence.db`** and letting it regenerate (still no migrations framework — see [database_notes.md](database_notes.md)). Verified with a real CLI run afterward: 8 platforms registered (2 available, 6 unsupported), 6 listings processed.
  - This directly narrows the long-open "which platform first" question in [../notes/Questions.md](../notes/Questions.md) from open-ended to a concrete shortlist.
