## 2026-07-14 — Version 2.0 Step 3: Search Memory & Comparison Engine

New `src/search_memory/` package: `models.py` (`SearchExecution`, `SearchComparison`, `SearchStatistics`, `SearchTimeline`, plus small `ApartmentPriceChange`/`ApartmentAvailabilityChange`/`PlatformCoverageChange` dataclasses), `comparison.py` (pure `diff_apartment_sets`/`platform_coverage_change`/`search_quality`), `search_memory_service.py` (write-side `record_completed_search`; read-side `latest_search`/`search_history`/`search_timeline`/`compare_searches`/`average_execution_time`/`average_apartment_count`/`search_statistics` — the mission's PascalCase method names translated to this project's snake_case). **A real bug found while proving this against the real pipeline, not just unit tests with hand-picked timestamps**: the originally-designed "changed since the previous run" check used a raw timestamp window (`previous.created_at < observed_at <= current.created_at`). This is wrong because a search's own initial-observation writes (a brand-new apartment's first price/title/image rows) happen strictly *after* that search's own `created_at` — processing (discovery, connector fetches, ranking, report generation) takes real time. Running the actual CLI twice back-to-back against unchanged data reported every apartment as "changed" on the second run, purely because the first run's own creation timestamp fell inside the naive window. Fixed by reconstructing "value as of search S" via `search_id` identity first (does this row belong to S itself, or was it written before S started?), falling back to a timestamp comparison only for rows outside the two searches being compared — sidesteps the processing-time gap entirely. Full writeup in `docs/17_Search_Memory.md` "A Real Bug" and `learning/architecture_notes.md`.

Verified against the real dev database, not just tests: ran the real CLI twice for the same location — the second run correctly reported 0 new/removed/changed apartments, with 12 `search_observed_apartments` rows accumulated across both runs (6 each, never merged or overwritten). Then edited `demo_platform`'s fixture price in place, ran a third time, and confirmed `search_memory_service.compare_searches()` reported the exact real price delta (1450 → 1399) before reverting the fixture. 156 tests passing (122 existing, untouched, + 34 new: comparison unit tests, service-level tests including the fixed run-over-run scenario and repeated-search/append-only regression tests, repository round-trip tests, and a new core-agent integration test file). Full detail in `learning/architecture_notes.md`.

## 2026-07-15 — Version 2.0 Step 5: Connector SDK & Plugin Framework

The largest single sprint of Version 2.0 — a full plugin framework, not just the template-method sketch drafted back when v2.0 was first designed. New `src/connectors/sdk/` package: `BaseConnector` (the template method — `connect -> fetch_listing -> parse -> normalize -> validate -> ConnectorResult`, implemented once, never overridden by a connector), `ConnectorFactory` (the only way `core/agent.py` obtains a connector — never imports or instantiates one directly), `ConnectorRegistry` (self-registration via `@register_connector`, resolved by importing `src.connectors.<name>` on first lookup — the registry itself never names a specific platform), `ConnectorMetadata`/`ConnectorCapabilities` (declarative coverage and capability discovery — `supports("images")`, `supports("coordinates")`, etc.), `ConnectorConfiguration` (per-instance runtime settings), `ConnectorValidator` (structured, warnings-not-errors-by-default field-completeness checks), and a `ConnectorException` hierarchy (`ConnectorConnectionError`/`ConnectorParsingError`/`ConnectorValidationError`/`ConnectorConfigurationError`).

`ConnectorHealth` was deliberately **not** redefined inside the SDK — it's `src.knowledge.models.ConnectorHealth` from v2.0 Step 4, re-exported and reused via `BaseConnector.health_check()` (which required adding a `platform_id` filter parameter to `knowledge_service.connector_health()`, backward-compatible, defaulting to the old all-platforms behavior). One `ConnectorHealth` concept, not two competing definitions of the same tracked facts.

Both reference connectors (`demo_platform.py`, `demo_platform_two.py`) were rebuilt on `BaseConnector`, removing the real, previously-duplicated fetch->save->parse sequence each independently implemented — every connector now implements exactly four small, genuinely platform-specific hooks: `build_url`, `parse` (raw response -> platform-native records), `normalize` (one record -> one `RawListing`), and `connector_info` (declares `ConnectorMetadata`). `core/agent.py`'s old `_load_connector`/`Connector` ABC were removed outright — nothing needed the `module.CONNECTOR` attribute convention once `ConnectorFactory` existed, and keeping both would have been redundant. `core/agent.py`'s per-platform loop got noticeably simpler too: timing/success/failure now comes from the `ConnectorResult` each connector returns (measured once, inside `BaseConnector.search()`) instead of `core/agent.py` measuring `time.perf_counter()` itself around every connector call, removing a second, redundant measurement.

A small, deliberate, zero-invention design choice: `RawListing` stays exactly where it's always been (`connectors/base.py`), completely untouched — every connector, regardless of source format (HTML today; a future JSON API, GraphQL, RSS, or CSV source), still produces this one shared shape, so nothing downstream of a connector (`analyzers/`, `ranking/`, `search_memory/`, `knowledge/`) needed to change at all. This is the concrete proof that "the Research Agent must never know whether data came from HTML, an API, or a future format" actually holds: not one line changed in any of those packages.

New `docs/18_Connector_SDK.md` — the mission asked for `docs/17_Connector_SDK.md`, but `17` was already taken by Search Memory (v2.0 Step 3); used the next free number and noted the discrepancy rather than silently renumbering anything. Covers architecture, lifecycle, how to build a new connector, best practices, and certification requirements. A new, reusable `tests/connectors/sdk/certification.py` mixin lets any connector's own test file certify SDK compliance (valid metadata, registered, queryable capabilities, returns a `ConnectorResult`, every listing has its required fields) without writing certification logic per connector — both reference connectors' test files now mix this in and pass it.

54 new tests (256 total: 202 existing untouched + 54 new): SDK unit tests for every new class, a template-method test suite using small scripted fake connectors (to test `BaseConnector`'s own orchestration and error-handling independent of any real platform's parsing logic), the certification suite applied to both reference connectors, and registry/factory performance tests proving lookup stays fast with hundreds of registered connectors. Verified against the real dev database, not just tests: ran the actual CLI through the new `ConnectorFactory -> BaseConnector` path end-to-end and confirmed identical results (6 apartments from the same 2 real connectors) plus correct Knowledge Engine observations, exactly matching pre-refactor behavior. ## 2026-07-16 — Version 2.5 Step 11: Intelligent Ranking Engine V2

Explicit instruction: "Transform apartment ranking from a static weighted score into a modular, explainable, evidence-based decision engine... The Ranking Engine must remain deterministic. No machine learning. No opaque scoring." Answered up front: (1) ranking is separated from analysis because analysis answers "what is objectively true," ranking answers "given this searcher's priorities, which apartments win" — separating them lets the same computed evidence serve many ranking profiles without recomputation; (2) explainability is non-negotiable because an opaque score is unactionable and undebuggable, and this project's whole discipline already rules out a black box; (3) every engine built since Step 4 (Dynamic Filters, Geographic Intelligence, Apartment History, Knowledge Engine, Provider Health, Search History, ...) is a candidate evidence source, per the mission's own INPUTS list; (4) confidence matters because evidence quality varies wildly (a curated fact vs. an honest "no evidence yet" default), and collapsing that into one score without tracking it would let weak evidence masquerade as strong.

New `src/ranking_v2/` package, mirroring `ConnectorRegistry`/`AnalysisRegistry`/`ProviderRegistry`/`FilterRegistry`/`GeoProviderRegistry`'s self-registration shape: `RankingRule` (both `evaluate()`/`metadata()` abstract — unlike `BaseFilter`/`BaseAnalyzer`'s "thin contract with defaults" shape, checked and deliberately rejected here, since the things a rule could do differently — read a database, read optional context, compute from nothing — genuinely differ enough per rule that no shared default would fit most of them), `RankingRuleRegistry`/`RankingWeights`/`RankingProfile`/`RankingPipeline`/`RankingEngineV2`/`RankingStatistics`. `RankingContext` is the widest context object in this codebase so far — one optional field per INPUT the mission names (`filter_results`/`geo_enrichments`/`analysis_results`/`provider_health`/`search_comparison`), plus `conn`/`location` for the inputs (Price via Knowledge Engine, Platform/Connector Reliability) that are always directly queryable given a connection.

12 built-in rules, one per named input, each reusing — never recomputing — a real, already-built engine's own read function: `price` (`knowledge_service.average_city_price()`, current price vs. city average, linear 0-100%-of-average-to-double scale), `price_trend` (`apartment_repository.get_price_history()`, needs 2+ observations), `walking_distance`/`public_transport` (`GeoEnrichment.distances[mode]`, reusing the Geographic Engine's own honestly-lower confidence for estimates), `availability` (`Apartment.current_status` — the one rule with unconditionally real evidence, confidence always 1.0), `lifestyle` (`GeoEnrichment.nearby`, confirmed categories only — an uncurated category is excluded from the average, never treated as zero), `filter_preferences`/`analysis_composite`/`provider_health`/`search_history` (all four honestly dormant unless the caller supplies the matching optional `RankingContext` field, the same "real, registered, tested, honest about missing evidence" pattern the Dynamic Filter Engine's dormant filters established, applied here to whole upstream engines rather than missing schema fields), `platform_reliability`/`connector_reliability` (`knowledge_service.platform_reliability()`/`connector_health()`, real once observations accumulate).

The core design decision, and the reason this engine can honestly call itself "evidence-based": per-apartment weight renormalization. A rule with no evidence for a given apartment is excluded from *both* the score numerator and the weight-normalization denominator for that apartment specifically — not counted as a zero. Verified directly with a controlled test (`tests/ranking_v2/test_pipeline.py::RenormalizationTests`): a 50/50-weighted profile where only one rule has evidence produces the *exact same* final score as a 100%-weighted profile on that one rule (80.0, not 40.0) — an apartment is never punished for missing optional context nobody asked this run to compute. `RankingProfile` ships two presets: `DEFAULT_PROFILE` (the mission's own worked example — Price 40%, Walking Distance 25%, Availability 15%, Public Transport 10%, Lifestyle 10%) and `COMPREHENSIVE_PROFILE` (every rule weighted equally) — `RankingWeights` accepts raw percentage-style values directly (`{"price": 40, ...}`) and normalizes them on read, matching the mission's own example notation exactly rather than forcing pre-normalized fractions.

Explanation building reuses each rule's own `detail` sentence, written in the mission's own qualitative tone ("Excellent walking distance," never `walking_distance_score=0.91`) via a small shared `qualitative()` phrase-bucket helper (`rules/_phrasing.py`) so 12 independently-written rules read as one consistent voice. Positive/negative factors are sorted by actual weighted contribution, not raw score, so the most decisive reasons surface first — matching the mission's own example ordering.

A real provider-independence-style violation was caught and avoided *before* writing the integration code, learning directly from Step 10's own post-hoc fix: `record_geo_enrichment()` (Step 10) originally took a hardcoded `provider_id` literal from `agent.py`, later fixed to derive it from the enrichment's own data. This sprint applied that lesson proactively — `RankingContext`/rules never assume which specific upstream engine instance produced their evidence, only its shape (`GeoEnrichment`, `AnalysisResult`, `ProviderHealth`), so no equivalent fix was needed here.

Integration follows the exact `data_router`/`ai_router`/`filter_engine`/`geo_engine` precedent: one new, optional, default-`None` `ranking_engine_v2` parameter on `RentalResearchAgent`, zero behavior change for any existing caller. Placed after v1 `RankingEngine.rank()` (which still does the actual hard-filtering and still writes `search_results.rank`/`.score` completely unchanged) — `RankingEngineV2` re-scores the same survivors with a real `RankingContext` (this run's own `conn`/`analysis_results`/`geo_enrichments`) and its output goes straight to `generate_report()` as an independent artifact, never wired into either ranking engine's own scoring — the same diagram-vs-implementation reconciliation now made a third time (Steps 6, 9, 10, 11), for the same reason each time: wiring a brand-new engine's output directly into an already-tested prior step's scoring risks destabilizing working code for no benefit the mission's own "REPORTS must display" section actually asks for. `filter_results`/`provider_health`/`search_comparison` are deliberately not auto-wired by `core/agent.py` this sprint — all three rules remain real, registered, and tested, just honestly dormant through the standard pipeline until a future sprint assembles that context, documented explicitly rather than silently incomplete.

## 2026-07-16 — Version 2.5 Step 12: User Feedback and Preference Learning Engine

Explicit instruction: "This sprint must not implement opaque machine learning. The platform must learn user preferences from explicit, traceable evidence. Application code remains deterministic... Do not modify ranking rules silently. Do not automatically change a user's saved preferences without recording why." Answered up front: (1) the Knowledge Engine observes system-wide platform facts, this engine observes one specific person's actions — different subjects, different lifecycles, conflating them would let personal taste pollute aggregate platform stats; (2) explicit feedback states intent with no interpretation gap, inferred behavior is a hypothesis about noisy action — implemented concretely as a `3×` weight multiplier in `decay.py`, not just a principle; (3) ranking influence flows through exactly one adapter module (`ranking_adapter.py`) that neither package imports the other around, producing only a *suggested* profile the user's explicit one always outranks unless `ASSISTED` mode is chosen; (4) two append-only tables (`feedback_events`, `preference_adjustments`) with no `update_*`/`delete_*` function anywhere — undo/reset write new rows, never touch old ones; (5) sensitive personal characteristics (gender, ethnicity, religion, health, sexual orientation, political views) are never inferred, enforced by a structural test checking every registered preference key/description/category against a blocklist, not just a docstring promise.

New `src/feedback/` package, mirroring `ConnectorRegistry`/`AnalysisRegistry`/`ProviderRegistry`/`FilterRegistry`/`GeoProviderRegistry`/`RankingRuleRegistry`'s self-registration shape for the 7th time: `PreferenceRule` (both `observe()` and `metadata()` abstract, but — unlike `RankingRule`'s "everything abstract" choice — `aggregate()` has a REAL shared default via four intermediate base classes, `ImportancePreferenceRule`/`ThresholdPreferenceRule`/`CategoricalPreferenceRule`/`BooleanPreferenceRule`). This was a deliberate, considered departure from Step 11's own precedent, not an inconsistency: the mission's "Learning Rules" section describes ONE consistent algorithm (decay, explicit-outweighs-inferred, conflict-reduces-confidence, single-action-damping) that every one of 23 dimensions must follow identically, the same kind of "one correct default for almost every case" that justified `BaseFilter`/`BaseAnalyzer`'s shared defaults back in Steps 6/9 — checked against the actual mission text before copying either prior pattern by default.

23 built-in preference rules, honestly split: 12 real, `Apartment`/`GeoEnrichment`-field-backed dimensions (`price_sensitivity` reusing `knowledge_service.average_city_price()`; `walking_distance`/`public_transport`/`lifestyle`/`nearby_services` reusing `GeoEnrichment` verbatim, including its own honestly-lower confidence for estimates; `property_type`/`minimum_area`/`number_of_rooms`/`platform` reading real `Apartment` fields; `availability_importance` reading `current_status`; `maximum_budget` a decayed-weighted average of positively-engaged prices, honestly documented as a proxy for "comfortable price point" rather than a literal maximum; `neighborhood` using the search `location` string, the only geography granularity this schema has), and 11 dormant-field dimensions (`room_type`, `private_bathroom`, `private_kitchen`, `air_conditioning`, `furnished`, `pets_allowed`, `balcony`, `parking`, `utilities_included`, `internet_included`, `number_of_flatmates`) — the identical "no structured schema field exists" situation `filter_engine/filters/amenities.py`'s 27 dormant filters already documented for these exact same fields (v2.5 Step 9). A saved/rejected apartment can never be checked for "does it actually have a balcony," so these 11 honestly learn only from explicit `FILTER_SELECTED`/`FILTER_REMOVED` events — real, structured evidence of intent, never corroborated against a listing outcome — reusing the identical `key` strings `filter_engine`'s own dormant filters already use for the same concepts, so a user's filter choice and the learned preference are one idea, not two.

Centralized decay/confidence math (`src/feedback/decay.py`), called by all four shared `aggregate()` bases rather than reimplemented per rule: `overall = consistency * volume` — `consistency` (0 fully conflicting to 1 fully one-directional) implements "conflicting behavior must reduce confidence"; `volume` (saturating at a configurable total weight, default 5.0) implements "a single action must not strongly alter the profile." Verified directly, not just reasoned about: a single inferred observation yields confidence ≤0.2, a single explicit one (weighted 3× by default) yields confidence ≤0.6 — meaningfully more trusted, never treated as certain either. `DecayConfig` (half-life, explicit multiplier, saturation weight) is a real constructor parameter on `FeedbackEngine`, not a hidden constant — "the decay rule must be configurable" is a dataclass field, not a promise.

The single hardest coherence problem this sprint solved: **making `undo_preference_adjustment()`/`reset_inferred_preferences()` genuinely effective rather than immediately overwritten by the next `build_preference_profile()` call.** If preference values were purely re-derived from the full observation history every time, an undo/reset would have no lasting effect — the very next rebuild would silently re-derive the identical value from the still-present (correctly still-present, per "never delete raw feedback events") historical observations, erasing the undo/reset within one call. Resolved by making the adjustment log itself the source of truth for "current" values: `build_preference_profile()` computes what the aggregate *would* be, but a reset/undo writes a new adjustment whose `applied_at` becomes a new evidence cutoff (`FeedbackEngine._effective_cutoff()`) — only observations *after* that cutoff feed future aggregation, so a manual undo/reset sticks until genuinely new events arrive, without ever touching a raw event or observation row. Verified directly: recorded one event, built a profile, reset it, rebuilt — the preference correctly reads as "no evidence" until a new event is recorded, while `export_feedback_history()` still shows the original event untouched.

A design decision made explicitly rather than defaulted to the more obvious choice: `maximum_budget`'s `ThresholdPreferenceRule` base computes a decayed-weighted *average* of positively-engaged prices, not a literal maximum — a single expensive outlier saved once wouldn't otherwise move a true max very informatively, and reusing the same shared aggregation base (rather than writing a bespoke "running max" aggregator just for this one dimension) kept the "one consistent algorithm" guarantee intact. Documented honestly in both the code and `docs/28` as a real, useful proxy rather than a literal reading of "maximum."

Ranking integration deliberately isolated to one module: `src/feedback/ranking_adapter.py` is the only file importing both `feedback` and `ranking_v2` — `FeedbackEngine` has zero awareness any `RankingRule` exists, and `RankingEngineV2` has zero awareness `feedback` exists, exactly "Do not couple FeedbackEngine directly to individual ranking rules" (the mission's own words). Three modes, verified with identity checks, not just value equality: `EXPLICIT_ONLY`/`SUGGESTED` both return the caller's own `RankingProfile` object completely unchanged (`is` identity), proving no side-channel mutation; only `ASSISTED` returns a new profile, seeded from the caller's own explicit weights as a base and overriding only the (currently 5) preference dimensions with a `ranking_v2` rule_key counterpart, and only above a confidence threshold (explicit preferences bypass it). The other 18 preferences don't yet have a matching ranking rule — an honest, documented limitation, the same "future modules" reasoning `docs/27`'s own section already used for a different gap.

Integration follows the exact `data_router`/`ai_router`/`filter_engine`/`geo_engine`/`ranking_engine_v2` precedent for the fifth time: three new, optional, default-`None`/`SUGGESTED` parameters (`feedback_engine`/`feedback_profile_id`/`feedback_mode`) on `RentalResearchAgent`, zero behavior change for any existing caller. A new, separate CLI entry point (`src/ui/feedback_cli.py`) rather than folding feedback commands into `ui/cli.py` — recording feedback/inspecting preferences is a genuinely different operation from running a search, and keeping the argument surfaces apart preserves `ui/cli.py`'s own backward-compatibility guarantee without either command's help text growing to accommodate the other.

130 new tests (864 total: 734 existing untouched + 130 new) — every new class's unit tests, all 23 rules' own `observe()` behavior (real evidence and honest no-evidence paths), migration + append-only-history tests (asserting no `update_*`/`delete_*` function exists), the mission's own explicit "Test that" list verified one by one (a single event never creates an extreme preference, repeated consistent events strengthen confidence, conflicting events reduce it, explicit settings override inferred evidence, missing listing fields never become negative signals, old feedback stays reproducible byte-for-byte, reset never deletes raw events), a plugin test, performance tests (500 events against all 23 real rules), a structural privacy guardrail test, agent-level integration tests (real Playwright-fixture pipeline, mocked at the `BrowserCollector` boundary), and CLI tests. Verified against a real, live run: built a neutral profile, ranked apartments, recorded explicit and inferred feedback, rebuilt the profile, re-ranked, and confirmed the ordering changed for an explainable reason while `EXPLICIT_ONLY` mode reproduced the original ordering exactly. None = None` parameter (every existing caller unaffected) so a saved search's
`selected_platforms`/`selected_connectors` allowlist and a policy's `enabled_
providers`/`disabled_providers` can genuinely narrow which connector-available
platforms get queried, at the query level — not just recorded as metadata nobody
enforces. `search_memory_service.py` gained one small additive public accessor,
`get_search_execution(conn, search_id)` — every other read function there was scoped
by `location`, and monitoring needed one specific prior run's `failed_platform_ids`
directly, not re-derived from a location-scoped history list.

Listing removal is a three-stage state machine (`missing` → `possibly_removed` →
`confirmed_removed`), gated by a configurable consecutive-miss threshold — "do not
mark a listing removed after one failed observation," the mission's own words —
firing `LISTING_REMOVED` exactly once, on the run that crosses the threshold, never
on every subsequent run it stays missing. `consecutive_absences()` takes an
already-batched, newest-first list of observed-apartment-id sets rather than
querying per apartment. Event deduplication was originally written *inside* each
detector's own event-building helper; refactored during this sprint to a single
central check in `MonitoringEngine`, after every detector has run — one
responsibility, one place, and it makes `MonitoringStatistics
.suppressed_duplicate_count` a trivial before/after count instead of five separate
counters. That refactor briefly left a stale `is_duplicate(...)` call inside
`RankingChangeDetector.detect()`'s main loop after the import was removed — a
`NameError` that would have crashed on the very first real rank-change or
better-match scenario in production. Caught by `tests/monitoring/test_detectors.py`,
not by manual smoke-testing, exactly the reason that test file was written before
moving on.

`monitoring_schedules` doubles as both "next run time" bookkeeping and the run-claim
lock: `claim_due_run()` is one atomic conditional `UPDATE ... WHERE claimed_by IS
NULL OR claim_expires_at < ?`, proven race-free with a dedicated `test_claim_
prevents_a_second_worker` test using two sequential calls with different worker ids
against the same schedule row.

## 2026-07-16 — Version 2.5 Step 15: Notification Delivery Engine

Explicit instruction: monitoring detects and creates events; notification
delivery separately sends eligible events through configured channels — keep
these responsibilities completely separate. NOT email/webhook/delivery logic
inside `MonitoringEngine`, NOT a web dashboard, NOT marketing/unsolicited
messaging, NOT SMS/mobile push/autonomous outbound messaging this sprint.

New `src/notifications/` package, a 10th (`NotificationChannel`) and 11th
(`NotificationTemplate`) application of this codebase's established self-
registration shape. Four channels ship this sprint: `console`/`file` (always
enabled, zero credentials), `email`/`webhook` (disabled by default, enabled
only once genuinely configured). Investigation before writing code confirmed
the "domain-layer dataclass distinct from storage-layer dataclass" pattern
(`monitoring/models.py` vs. `storage/models.py`) and the "Protocol + one real
implementation" injectable-seam pattern (`discovery.automatic.verification
.PageFetcher`) both extend cleanly into notifications — `EmailTransport`/
`HttpTransport` mirror `PageFetcher` directly, so the test suite never opens a
real SMTP connection or sends a request to a real endpoint.

The mission's own ELIGIBILITY section lists quiet-hours/rate-limits as
eligibility factors, but its own WORKFLOW diagram lists "Evaluate eligibility"
and "Apply quiet hours and rate limits" as two separate steps. Resolved by
splitting responsibility: `eligibility.py` handles content-based checks only
(preference enabled, event type/severity/significance, channel availability);
`quiet_hours.py`/`rate_limiting.py` handle the time-dependent, deferral-
capable checks, applied by `NotificationEngine` as its own explicit
subsequent step — documented directly in `eligibility.py`'s own module
docstring so the split doesn't look accidental later.

Digest scheduling was deliberately *not* given its own `notification_schedules`
table — `next_digest_time()` derives "next due" from the most recent
`notification_digests.period_end` row plus a frequency interval, avoiding a
redundant schema addition the mission's own DATABASE section didn't
literally require (it names `digest schedule` as an index target, which
`notification_deliveries`'s existing indexes already satisfy for a digest-type
delivery). Similarly, one `notification_delivery_events` link table serves
both immediate and digest deliveries — mission text said "digest-event
links," but generalizing to "every delivery's event links" was the same table
shape without inventing a second, parallel one.

Mid-session bug hunts, both caught by the test suite rather than manual
smoke-testing: (1) `get_due_retries()` originally selected only
`status = 'retry_scheduled'`, missing `partially_delivered` deliveries that
still have failed channels worth retrying — "one channel failure does not
prevent another channel from succeeding" applies symmetrically to retries,
not just first attempts; (2) `smtplib.SMTPException` subclasses `OSError` in
this Python version, so the email channel's original except-clause ordering
(`except (SMTPConnectError, SMTPServerDisconnected, TimeoutError, OSError)`
before `except SMTPException`) made the `"server_error"` category
unreachable for any generic SMTP exception — every one of them was being
caught by the earlier `OSError` clause and miscategorized as
`"connection_error"`. Found while writing
`test_generic_smtp_exception_is_categorized_as_server_error`, fixed by
reordering except clauses most-specific-first and adding a comment
explaining why the ordering matters (the fix would silently regress if a
future edit re-ordered them back).

Retries are idempotent by construction: `NotificationDelivery
.idempotency_key` is stable per (preference, event) or (preference, digest
period); `retry_due_failures()`/`retry_delivery_now()` always resolve and
reuse the same delivery row, computing "already delivered" channels from
`NotificationAttempt` history so a channel that already succeeded is never
re-sent — proven directly by
`test_retrying_is_idempotent_and_only_reattempts_channels_not_yet_delivered`
(one always-succeeding channel + one always-failing test-only channel,
retried twice, asserting exactly one attempt row for the former and two for
the latter).
