# 03 — Data Model

Status: V1.0 schema confirmed (2026-07-14). Storage engine: SQLite, single file at `data/rental_intelligence.db` (see [01_System_Architecture.md](01_System_Architecture.md) for why).

## The Versioning Principle, Concretely

Principles 1, 3, and 4 from [00_Project_Vision.md](00_Project_Vision.md) ("never lose information," "historical versions," "reproducible searches") translate into one concrete rule that shapes every table below:

**`apartments` holds current state. History tables are append-only and never updated or deleted. `search_results` is an immutable snapshot.**

Without that third part, principle 4 breaks silently: if a report from three months ago just joins against the live `apartments` table, its numbers change every time you look at it again as prices update — it stops being "that search, as it was." So `search_results` denormalizes (copies) the price/status *as observed at that search*, rather than only storing a foreign key to the current apartment row.

## Entities

### `platforms`

The Platform Registry, managed by the Multi-Platform Discovery Framework (see
[05_Platform_Discovery.md](05_Platform_Discovery.md)). As of v1.1 this table records
**every rental platform the system knows about**, whether or not a connector exists for
it yet — `connector_available` is what distinguishes "known" from "usable."

| Column | Type | Notes |
|---|---|---|
| `id` | TEXT PK | Stable slug, e.g. `"demo_platform"` — this is "platform_id" |
| `name` | TEXT | Human-readable |
| `country` | TEXT | — |
| `supported_cities` | TEXT (JSON list) | e.g. `["Nationwide"]` if not verified down to specific cities |
| `rental_types` | TEXT (JSON list) | What the platform itself covers (may be broader than this system currently processes — see [00_Project_Vision.md](00_Project_Vision.md) rental-type scope) |
| `homepage` | TEXT | Public homepage URL |
| `search_url` | TEXT, nullable | URL template used to build search queries, once known |
| `requires_login` | INTEGER (bool) | Whether browsing/search requires an account |
| `connector_available` | INTEGER (bool) | Whether a working `Connector` implementation exists for this platform — see [06_Connector_Framework.md](06_Connector_Framework.md). `False` means "known, catalogued, not yet supported" |
| `connector_name` | TEXT, nullable | Matches a module under `connectors/` (e.g. `"demo_platform"` → `connectors.demo_platform`), null until a connector exists |
| `last_verified` | TEXT (ISO 8601), nullable | When this platform's metadata was last confirmed accurate; null if never independently verified |
| `discovery_method` | TEXT | How this platform entry was found — e.g. `"manual_research"`, `"manual_seed"` (V1.1); `"web_search"` reserved for future automated discovery |
| `notes` | TEXT, nullable | Free text — auth quirks, rate limits, caveats about unverified fields, etc. |
| `created_at` | TEXT (ISO 8601) | — |

**Dropped from v1.0:** `base_url` (split into `homepage`/`search_url`, which are genuinely different things) and `is_active` (superseded by `connector_available` — a platform with no connector can't be searched regardless of any separate "active" flag, so a second boolean added no real distinction; see [05_Platform_Discovery.md](05_Platform_Discovery.md)).

### `apartments`

Current state, one row per (platform, platform's own listing ID). See [07_Analysis_Engine.md](07_Analysis_Engine.md) for how this gets written.

| Column | Type | Notes |
|---|---|---|
| `id` | TEXT PK | Synthetic ID (UUID), generated by us — stable identity even if the platform's own ID scheme is unstable |
| `platform_id` | TEXT FK → `platforms.id` | — |
| `platform_listing_id` | TEXT | The ID/slug from the source site |
| `title` | TEXT | — |
| `bedrooms` | REAL, nullable | — |
| `bathrooms` | REAL, nullable | — |
| `sqft` | REAL, nullable | Nullable — not every platform reports it |
| `address_raw` | TEXT | As scraped |
| `address_normalized` | TEXT (JSON), nullable | Structured form once the Analysis Engine can produce one — *TBD normalization rules* |
| `latitude` / `longitude` | REAL, nullable | For distance-based ranking criteria |
| `url` | TEXT | Original listing URL (Principle: "original listing URLs") |
| `current_price` | REAL | Monthly rent, normalized currency/period |
| `current_status` | TEXT | e.g. `available`, `pending`, `delisted` — see `apartment_availability_history` for the trail that produced this |
| `first_seen_at` | TEXT (ISO 8601) | Set once, never updated |
| `last_seen_at` | TEXT (ISO 8601) | Updated every time a search re-observes this listing |
| `merged_into_id` | TEXT FK → `apartments.id`, nullable | **Unused in V1**, reserved for V2 cross-platform de-duplication (see [07_Analysis_Engine.md](07_Analysis_Engine.md)) so that feature doesn't require a schema migration to add the column later |

**Unique constraint:** (`platform_id`, `platform_listing_id`) — this is the V1 identity rule. The same physical apartment listed on two platforms gets two rows; merging them is explicitly V2 (see [00_Project_Vision.md](00_Project_Vision.md) Non-Goals).

| Column | Type | Notes |
|---|---|---|
| `id` | INTEGER PK AUTOINCREMENT | — |

## Entity Relationship Summary

```
platforms 1──* apartments 1──* apartment_price_history
                          │ 1──* apartment_availability_history
                          │ 1──* apartment_images
                          │ *──* search_results ──* search_requests
                          │
                          *──* raw_captures ──* search_requests

knowledge_entries — standalone, referenced by analyzers/ranking logic, not FK-linked
```

## Storage Format Outside SQLite

Confirmed folder layout under `data/` (see [02_Folder_Guide.md](02_Folder_Guide.md)): `apartments/` and `search_history/` as originally scaffolded are **superseded by the SQLite tables above** — the database file `data/rental_intelligence.db` is now the source of truth for that data, not per-record files in those folders. `platform_registry/` similarly becomes redundant with the `platforms` table. `media/` and `raw_pages/` remain file-based (referenced by path from `apartment_images`/`raw_captures`). `knowledge_base/` remains available for hand-authored source material that later gets loaded into `knowledge_entries`. `cache/` stays a plain file cache, outside the database (Principle: it's explicitly not a source of truth).
