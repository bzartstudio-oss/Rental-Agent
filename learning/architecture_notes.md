# Architecture Notes

Architecture decisions and project-structure lessons for this project. See [Project Learning.md](Project%20Learning.md) for the full index.

## Decisions

- **Canonical project location is `Clients/Rental-Agent`.** An earlier duplicate scaffold at `Clients/Agent Rental/Rental-Agent` was created by mistake and has been removed — its content had already been moved into the real folder before that was noticed.
- **2026-07-13 — Rental type decided: residential apartments.** Confirmed when creating the `data/apartments/` subfolder made the implicit choice visible and it was checked against the user rather than assumed. Vacation/short-term and equipment rentals are out of scope unless revisited. This closed the open question in [../notes/Questions.md](../notes/Questions.md) and updated `Listing` fields in [../docs/03_Data_Model.md](../docs/03_Data_Model.md) to residential-specific ones (bedrooms, bathrooms, sqft).

Whether the generated/large ones (`cache/`, `raw_pages/`, `media/`, and possibly `apartments/`/`search_history/`) should be gitignored is explicitly deferred until there's real data to gauge size against — tracked in [../notes/Questions.md](../notes/Questions.md), don't assume an answer before then.
- **2026-07-13 — Established a documentation-first workflow via a numbered `docs/` structure (`00_Project_Vision.md` through `14_Lessons_Learned.md`).** Rationale: several core decisions (target rental type, data source, ranking criteria) weren't made yet, so the docs were written as skeletons with explicit `*TBD*` markers rather than invented content — this gives every future implementation piece a defined place to live and a place to record its "why" without pretending unmade decisions are settled. ## Lessons Learned / Known Issues

- **Known issue, deliberately not fixed yet: `src/` has pre-existing folders that don't map onto the structure above** — `ai/`, `browser/` (has `browser_manager.py`), `config/` (has `config_loader.py`), `data_source/`, `exporters/`, `filters/`, `maps/`, `model/` (has `apartment.py`), `models/` (has `configuration.py`), `reports/`. `model/` vs `models/` in particular looks like an accidental duplicate. Explicitly deferred — reconcile (map old code into `connectors/`, `storage/`, etc., or delete duplicates) before or while writing new code in the overlapping areas, so logic doesn't end up split across both an old and a new folder for the same concern.

Also found (older, from 2026-07-12, likely from an earlier untracked work session, not this incident): `docs/projectblueprint.md`, `docs/developmentroadmap.md`, `docs/developmentjournal.md` — all empty, functionally superseded by `docs/00_Project_Vision.md`, `docs/10_Roadmap.md`, `docs/11_Development_Journal.md`.
  - **Investigated and ruled out:** no scheduled task (`list_scheduled_tasks`) or cron job (`CronList`) was configured against this project; Desktop is not OneDrive-redirected (checked `HKCU\...\User Shell Folders` and reparse-point status) so it isn't a OneDrive sync conflict.
  - **Root cause: undetermined.** The clustering of all changes within about a minute, uniformly empty content, and a mix of generic (`001_Decision.md`) and oddly-specific-to-this-project filenames is most consistent with a scripted "ensure these files exist" scaffold tool (e.g. something using `New-Item -Force`-style semantics) run once against this folder — not a live/ongoing process. The user confirmed no other session should have been running. **Recommend checking (not yet checked):** other open Claude Code / AI-tool windows or browser tabs pointed at this same folder, any recently-run IDE "generate project" extension, and Windows' own Task Scheduler (distinct from the `scheduled-tasks`/`CronList` tools checked above, which only cover tasks created through this harness).

The unexplained empty leftovers were deleted; `src/browser/__init__.py` (pre-existing before this incident, just now also empty) was left alone.

  - **2026-07-14 addendum:** more casualties from the same incident surfaced later — empty `README.md` files had also been added to `output/` and every `src/` module folder (`analyzers/`, `collectors/`, `core/`, `discovery/`, `ranking/`, `reports/`, `services/`, `storage/`, `ui/`), plus `src/connectors/README.md` (which had real content, written 2026-07-13) was wiped like the `docs/` files. `src/connectors/README.md` was restored from conversation history and updated for the V1.0 design below; the other empty, unexplained `README.md` files were deleted. - **2026-07-14 — V1.0 architecture design session: "Rental Intelligence Platform."** The project was reframed from a simple scraper to a self-improving research platform, per 7 core principles now recorded in [../docs/00_Project_Vision.md](../docs/00_Project_Vision.md) (never lose information, every search updates permanent databases, historical versions, reproducible/comparable searches, configurable/extensible filters, extensible to new platforms/cities without redesign, business logic independent of any website). Every numbered doc from `00` through `10` was rewritten to a concrete V1.0 design (not just skeletons) in one session. Key decisions:
  - **Storage engine: SQLite** — see [database_notes.md](database_notes.md) for the full reasoning. This supersedes the plain-file `data/apartments/`, `data/search_history/`, `data/platform_registry/` folders decided 2026-07-13 (not deleted, just no longer the write target — see [../docs/02_Folder_Guide.md](../docs/02_Folder_Guide.md)).
  - **Connectors vs. Collectors split** introduced: `connectors/` holds only platform-specific query/parsing logic; `collectors/` holds generic, reusable fetch infrastructure (browser, HTTP, image download, raw-page persistence) that connectors call into. This is what the Principle-7 "no website-specific logic outside connectors" boundary actually rests on — without it, browser/fetch mechanics would keep getting duplicated per connector. See [../docs/06_Connector_Framework.md](../docs/06_Connector_Framework.md).
  - **Immutable snapshot design for `search_results`**: denormalizes price/status at search time rather than joining to live data, so old reports don't silently change as new searches update `apartments`. See [database_notes.md](database_notes.md).
  - **Legacy `src/` folder reconciliation finally resolved** (deferred since 2026-07-13, see above): concrete disposition for every legacy folder (`browser/` → `collectors/browser_collector.py`, `config/` → `core/config.py`, `model/`/`models/` → `storage/` + delete the duplicate, `data_source/`/`filters/`/`exporters/`/`reports/`/`maps/` → delete, `ai/` → keep, reserved for V2). Table lives in [../docs/02_Folder_Guide.md](../docs/02_Folder_Guide.md); execution is Roadmap Phase 1, not done yet.
  - **Cross-platform apartment de-duplication deliberately deferred to V2** — the schema reserves a `merged_into_id` column now so V2 doesn't need a migration, but V1 only de-duplicates within a single platform. See [../docs/07_Analysis_Engine.md](../docs/07_Analysis_Engine.md).
  - **Extensibility is deliberately asymmetric**: new platforms are a first-class, zero-redesign case (the whole point of the Connector contract); new rental types are *not* pre-built into a generic schema, only documented as an upgrade path — building that generalization now, before a second rental type is ever requested, would violate the project's own "don't design for hypothetical requirements" rule. See "Extensibility Without Over-Engineering" in [../docs/01_System_Architecture.md](../docs/01_System_Architecture.md).
  - Full V1.0 build order, phased with explicit exit criteria per phase, in [../docs/10_Roadmap.md](../docs/10_Roadmap.md) — starts with the storage layer (Phase 1) since every other module depends on it, ends with a second connector (Phase 7) specifically to validate the platform-independence boundary holds under real pressure, not just on paper.

- **2026-07-14 — Version 2.0 architecture design session: "Autonomous Rental Intelligence Platform."** User instruction, explicit: architecture only, no `src/` code changes, no connectors, until the design is complete — this session produced only `docs/` changes, verified via `git status` before committing. Full design across [../docs/00](../docs/00_Project_Vision.md) (8th principle added: learning through data, never by rewriting code), [../docs/01](../docs/01_System_Architecture.md) (feedback-loop diagram), [../docs/03](../docs/03_Data_Model.md) (schema — 6 new tables, ~18 new nullable columns across 4 existing tables), [../docs/04](../docs/04_Search_Request.md)–[../docs/07](../docs/07_Analysis_Engine.md) (extended), and three new docs (`15_Agent_Architecture.md`, `16_Knowledge_Engine.md`, `17_Search_Memory.md`). Key decisions:
  - **Generic append-only patterns over per-field tables**: `apartment_change_log` (field_name/old_value/new_value) catches title/description/future-field changes instead of a dedicated table per field; `apartment_analysis_metrics` (metric_name/metric_value) catches every Deep Analysis Engine output instead of a column per metric. Both exist specifically so "future X must be addable without changing existing code" is literally true at the schema level, not just the code level.
  - **`filter_definitions` is metadata only, not a rule engine.** Explicitly rejected building a data-driven expression/rule interpreter for filters — the requirement is "addable without changing *existing* code" (satisfied by the existing open/closed `FilterDefinition` registry, reorganized into `search/filters/` by category), not "addable with *no* code," which would be substantial unrequested scope creep.
  - **Room/flatshare filter categories (gender, room_type, private_bathroom, student_only) are designed but not resolved as in-scope** — these were flagged as reintroducing the exact concept confirmed stale from the old `config/settings.json` (see the 2026-07-14 entry above). Handled via `filter_definitions.applicable_rental_types_json` so the tension is architecturally harmless either way, without silently deciding the underlying product-scope question.
  - **First real migrations mechanism** (`storage/migrations/`, a `schema_migrations` tracking table) — v1.1's schema change was breaking (columns dropped/renamed) and required a dev-db reset; v2.0's is entirely additive (new tables, new nullable columns), so it's the first schema change that can safely run against a database with real accumulated data instead of resetting it. This is the "first real schema change" moment predicted (and deliberately not pre-built for) back in Phase 1.
  - **SQLite reconsidered, not replaced**, against "hundreds of thousands of records, multiple countries" — concluded the choice still holds (SQLite handles single-digit millions of rows fine); what would actually invalidate it is concurrent multi-writer access or true distributed deployment, neither of which is a real v2.0 requirement. Documented as a deliberate re-confirmation, not a default.
  - **`RentalResearchAgent` is explicitly NOT being split into per-agent objects yet** — [15_Agent_Architecture.md](../docs/15_Agent_Architecture.md) maps all 6 named agents onto existing or designed code (4 of 6 already have real logic, just invoked as function calls rather than agent objects) and documents the future split (a `PipelineContext` shape) without doing it now, since forcing that abstraction before a third independent agent actually needs it would be premature.
  - Full migration plan and dependency-ordered implementation sequence (schema → apartment history → search memory → knowledge engine → connector SDK → filter engine → deep analysis engine, deep analysis last because it's blocked on an unmade geocoding/transit vendor decision) in [../docs/10_Roadmap.md](../docs/10_Roadmap.md) "Version 2.0".

- **2026-07-14 — Version 2.0 Step 2: Apartment History Engine (122 tests passing, up from 79).** Scoped to exactly Step 1's follow-on per explicit instruction — the Migration Framework's schema (`apartment_change_log`, `apartment_image_events`) now gets real read/write logic; Search Memory and Knowledge Engine business logic remain explicitly out of scope. Key decisions:
  - **New `src/history/` package, not folded into `analyzers/`.** `models.py` (`Change`/`ChangeType` — the uniform, structured comparison result the mission asked for), `comparison.py` (pure, DB-free per-field comparison functions), `history_service.py` (write-side `record_new_apartment`/`record_reobservation` + read-side `latest_version`/`previous_version`/`price_timeline`/`availability_timeline`/`change_timeline`, as plain functions — no class, since there's no state beyond the `conn` every call already takes, matching `discovery/platform_registry.py`'s style rather than the stateful `*Agent` classes).
  - **Deliberate deviation from [../docs/07_Analysis_Engine.md](../docs/07_Analysis_Engine.md)'s original plan**: that doc proposed extending `analyzers/change_detector.py` with `title_changed`/`description_changed` booleans. Implemented instead as self-contained comparison functions in `src/history/comparison.py` (each does its own equality check and returns a `Change` or `None`), so every one of the new v2.0 fields — not just the two scalar ones — produces the same structured object, including `images` (a list-diff) and `presence` (not a simple equality at all). `change_detector.py` itself is untouched; price/status keep writing to their pre-existing dedicated history tables exactly as before, unrelated to the new generic `apartment_change_log` path.
  - **`_collect_images` replaced by one unified `_sync_images`** (`analyzers/engine.py`) used for both branches: a brand-new apartment has no current images yet, so the same diff logic naturally reports every URL as "added," in original listing order — proven behavior-identical to V1.0 by keeping the existing image tests passing unmodified. Re-observation image diffing (download new, flip removed to `is_current=0`, log both to `apartment_image_events`) is the actual new capability.
  - **`apartment_image_events.search_id` is `NOT NULL`** (an image event is always tied to the run that detected it) — event logging (not the download/flip) is skipped when `process_listing()` is called without a `search_id`, which only happens in direct unit tests; the real `RentalResearchAgent` always has a real `search_requests.id` by the time it calls this. Discovered while designing the new-apartment path, not by a test failure.
  - **`compare_coordinates` and `compare_presence` implemented and unit-tested but not wired into the pipeline** — coordinates aren't populated by any connector yet (waits on the Deep Analysis Engine, Step 7); "listing removed"/"listing returned" require Search Memory's full-observed-set comparison (Step 3) to mean what the mission intends, not a status-string heuristic. Building the comparison function now and wiring it in later, once its real trigger exists, avoided inventing pipeline behavior beyond what's actually driven by real data.
  - **`previous_version()` is a per-field reconstruction, not a single snapshot lookup** — there is no unified "apartment version" table in the schema (by design, see [../docs/03_Data_Model.md](../docs/03_Data_Model.md)), so "the state before the latest change" is assembled field-by-field from whichever history table each field lives in (second-newest row for price/status, the latest `apartment_change_log` entry's `old_value` for title/description).
  - **Verified against the real dev database, not just tests**: edited `demo_platform`'s fixture title in place, ran the actual CLI, confirmed a real `apartment_change_log` row with the correct old/new values and a real `search_id`, then reverted the fixture and ran again, confirming the reversion recorded as a second row without disturbing the first — the append-only guarantee proven live, not just asserted in a temp-file test.

- **2026-07-14 — Version 2.0 Step 3: Search Memory & Comparison Engine (156 tests passing, up from 122).** Scoped to exactly Step 3 — Search Memory and run-over-run comparison; explicitly no Knowledge Engine logic, no AI/predictions. Key decisions:
  - **New `src/search_memory/` package, mirroring `src/history/`'s shape**: `models.py` (`SearchExecution`/`SearchComparison`/`SearchStatistics`/`SearchTimeline` — the mission's four named models — plus small `ApartmentPriceChange`/`ApartmentAvailabilityChange`/`PlatformCoverageChange` dataclasses), `comparison.py` (pure, DB-free set/arithmetic functions), `search_memory_service.py` (plain functions again, not a class — same reasoning as `history_service.py`). New `storage/search_memory_repository.py` for `search_observed_apartments` and the `search_requests` completion `UPDATE`.
  - **`SearchExecution.pdf_report_path`/`warnings` are always empty/`None` today, deliberately** — no PDF Report Generator exists in this codebase (HTML is the only format, see [../docs/09_Report_System.md](../docs/09_Report_System.md)) and nothing in the pipeline currently detects a "warning-worthy" condition. Both fields live inside `runtime_stats_json` (the documented free-form escape hatch, [../docs/03_Data_Model.md](../docs/03_Data_Model.md)) rather than new schema columns — no migration needed, and the slots are ready the moment either feature is built, matching the same "dormant field" pattern as `platforms.connector_version` since v2.0 Step 1.
  - **A critical bug, found only by testing against the real pipeline, not hand-picked unit-test timestamps**: the originally-designed comparison (`docs/17_Search_Memory.md`) bounded "did this apartment change" by a raw timestamp window — `previous.created_at < observed_at <= current.created_at`. This is broken because `SearchRequest.created_at` is stamped at *request construction*, before any processing happens, while a search's own writes (a brand-new apartment's first price/title/image rows) land strictly *after* that timestamp, however long discovery/fetching/ranking/report-generation actually took. Running the real CLI twice back-to-back against completely unchanged fixture data reported **every single apartment as "changed"** on the second run — the first run's own creation-time writes fell inside the second run's comparison window purely because of normal processing latency. My own unit tests didn't catch this because they used a single shared timestamp variable for both "when the search was created" and "when the apartment was observed," accidentally eliminating the exact gap that breaks in production.
    - **Fix**: reconstruct "the value of field X as of search S" by matching on `search_id` identity first (`entry.search_id == S.id`), falling back to `entry.observed_at <= S.created_at` only for rows that don't belong to S itself (i.e., earlier searches, or history written outside any search context). This sidesteps the processing-time gap entirely since it doesn't matter *when* S's own rows were written relative to `S.created_at` — they're unambiguously S's, by id.
    - **Lesson**: a time-window design for comparing two runs needs a test that actually measures real wall-clock processing time between "request created" and "request's own writes happen" — not a unit test where both are conveniently pinned to the same variable. Verified the fix for real: ran the actual CLI twice for the same location (second run correctly reported 0 new/removed/changed against 12 real accumulated `search_observed_apartments` rows), then edited `demo_platform`'s fixture price and ran a third time, confirming `compare_searches()` reported the exact real delta (1450 → 1399) before reverting the fixture. Full writeup in [../docs/17_Search_Memory.md](../docs/17_Search_Memory.md) "A Real Bug".
  - **`analyzers/engine.py` gained its first Step 3 write**: a `search_observed_apartments` row per processed listing, guarded by `if search_id` exactly like `apartment_image_events` (the column is `NOT NULL` too) — Step 2 deliberately didn't touch this table since it was out of that step's scope.
  - **`core/agent.py`'s `except Exception: continue` now captures the exception message** (`errors.append(f"{platform.id}: {exc}")`) before continuing — a small, well-justified addition since Search Memory needs real error data for `runtime_stats`, and the information was already available right there and previously just discarded. Does **not** touch `platform_performance_observations` or any Knowledge Engine table — that recording is explicitly Step 4's job, not this one's.
