## 2026-07-14 — Sprint V2.0.1: Migration Framework

First implementation step of Version 2.0, scoped to exactly Step 1 of the implementation order — migration framework and schema only, explicitly no Apartment History/Knowledge Engine business logic yet. Added `storage/migrations/` and a `schema_migrations` tracking table; `storage/database.py` now discovers, orders (numerically, not alphabetically), and applies migrations after `schema.sql`, each in its own real transaction. `0001_v2_knowledge_engine.sql` contains exactly the schema from `docs/03_Data_Model.md` — 6 new tables, ~18 new nullable columns across `platforms`/`apartments`/`apartment_images`/`search_requests` (corrected two off-by-one column-count typos in `docs/10_Roadmap.md`'s summary along the way: platforms is +6 not +7, search_requests is +9 not +8 — `docs/03` was always the correct source). Updated the four affected dataclasses and their repository read/write code.

Found and fixed a real bug while writing the failed-migration-rollback test: Python's `sqlite3` module doesn't implicitly open a transaction before DDL (`CREATE`/`ALTER`) the way it does for DML, so a failed migration's earlier `CREATE TABLE` was committing immediately regardless of the wrapping `rollback()` call — the happy-path tests wouldn't have caught this, only the deliberately-broken-migration test did. Fixed with an explicit `BEGIN`/`COMMIT`/`ROLLBACK` transaction for the migration runner specifically (`Database.transaction()`, used everywhere else, is unaffected — it only ever wraps DML). Full writeup in `learning/python_notes.md`.

Proved the core promise for real, not just in tests: ran the actual CLI against the real `data/rental_intelligence.db` (which had genuine v1.1 data — 8 platforms from earlier real runs) and confirmed it migrated in place with zero data loss, no reset required. 79 tests passing (73 existing, untouched, + 6 new migration tests). Created `CHANGELOG.md` (didn't exist before this sprint).

## 2026-07-14 — Version 2.0 Step 2: Apartment History Engine

Second implementation step of Version 2.0, scoped to exactly Step 2 of the implementation order (`docs/10_Roadmap.md`) per explicit instruction — the Migration Framework's schema (`apartment_change_log`, `apartment_image_events`) now gets real read/write logic; Search Memory and Knowledge Engine business logic remain explicitly out of scope for this sprint. New `src/history/` package: `models.py` (`Change`/`ChangeType`), `comparison.py` (pure per-field comparison functions — price, availability, title, description, coordinates, images, listing presence, plus a "listing updated" summary), `history_service.py` (write-side `record_new_apartment`/`record_reobservation` plus read-side `latest_version`/`previous_version`/`price_timeline`/`availability_timeline`/`change_timeline`). New `storage/apartment_history_repository.py` for the two generic history tables' data access. `RawListing`/`normalizer.py` gained `description`; `apartment_repository.py` gained `update_apartment_details` and `mark_image_not_current`. `analyzers/engine.py`'s write sequence now also writes generic change-log rows for title/description and runs Image Change Detection (one unified `_sync_images` function used for both new and re-observed apartments) — exactly the write sequence documented in `docs/07_Analysis_Engine.md`.

One deliberate deviation from that doc's original plan: title/description comparison logic was consolidated into `src/history/comparison.py` rather than added to `analyzers/change_detector.py` as sketched, so every new v2.0 field — including images (a list-diff) and presence (not a scalar comparison at all) — produces the same structured `Change` object uniformly. `change_detector.py` itself is untouched. `compare_coordinates` and `compare_presence` ("listing removed"/"listing returned") are implemented and unit-tested but deliberately not wired into the pipeline yet — coordinates aren't populated by any connector (Step 7), and "removed" needs Search Memory's full-observed-set comparison (Step 3) to mean what the mission intends rather than a status heuristic.

Verified against the real dev database, not just tests: edited `demo_platform`'s fixture title in place, ran the actual CLI, confirmed a real `apartment_change_log` row appeared with the correct old/new values, then reverted the fixture and ran again, confirming the reversion recorded as a second row without disturbing the first — the append-only guarantee proven live. 122 tests passing (79 existing, untouched, + 43 new: comparison unit tests, history service tests including a reconstructed-previous-version test and a 500-row performance test, repository round-trip tests, and engine-level regression/integration tests). Full detail in `learning/architecture_notes.md`.
