## 2026-07-14 — Version 2.0 Step 3: Search Memory & Comparison Engine

Third implementation step of Version 2.0, scoped to exactly Step 3 — Search Memory and run-over-run comparison; explicitly no Knowledge Engine logic, no AI/predictions. New `src/search_memory/` package: `models.py` (`SearchExecution`, `SearchComparison`, `SearchStatistics`, `SearchTimeline`, plus small `ApartmentPriceChange`/`ApartmentAvailabilityChange`/`PlatformCoverageChange` dataclasses), `comparison.py` (pure `diff_apartment_sets`/`platform_coverage_change`/`search_quality`), `search_memory_service.py` (write-side `record_completed_search`; read-side `latest_search`/`search_history`/`search_timeline`/`compare_searches`/`average_execution_time`/`average_apartment_count`/`search_statistics` — the mission's PascalCase method names translated to this project's snake_case). **A real bug found while proving this against the real pipeline, not just unit tests with hand-picked timestamps**: the originally-designed "changed since the previous run" check used a raw timestamp window (`previous.created_at < observed_at <= current.created_at`). This is wrong because a search's own initial-observation writes (a brand-new apartment's first price/title/image rows) happen strictly *after* that search's own `created_at` — processing (discovery, connector fetches, ranking, report generation) takes real time. Running the actual CLI twice back-to-back against unchanged data reported every apartment as "changed" on the second run, purely because the first run's own creation timestamp fell inside the naive window. Fixed by reconstructing "value as of search S" via `search_id` identity first (does this row belong to S itself, or was it written before S started?), falling back to a timestamp comparison only for rows outside the two searches being compared — sidesteps the processing-time gap entirely. Full writeup in `docs/17_Search_Memory.md` "A Real Bug" and `learning/architecture_notes.md`.

Verified against the real dev database, not just tests: ran the real CLI twice for the same location — the second run correctly reported 0 new/removed/changed apartments, with 12 `search_observed_apartments` rows accumulated across both runs (6 each, never merged or overwritten). Then edited `demo_platform`'s fixture price in place, ran a third time, and confirmed `search_memory_service.compare_searches()` reported the exact real price delta (1450 → 1399) before reverting the fixture. 156 tests passing (122 existing, untouched, + 34 new: comparison unit tests, service-level tests including the fixed run-over-run scenario and repeated-search/append-only regression tests, repository round-trip tests, and a new core-agent integration test file). Full detail in `learning/architecture_notes.md`.

## 2026-07-15 — Version 2.0 Step 5: Connector SDK & Plugin Framework

The largest single sprint of Version 2.0 — a full plugin framework, not just the template-method sketch drafted back when v2.0 was first designed. New `src/connectors/sdk/` package: `BaseConnector` (the template method — `connect -> fetch_listing -> parse -> normalize -> validate -> ConnectorResult`, implemented once, never overridden by a connector), `ConnectorFactory` (the only way `core/agent.py` obtains a connector — never imports or instantiates one directly), `ConnectorRegistry` (self-registration via `@register_connector`, resolved by importing `src.connectors.<name>` on first lookup — the registry itself never names a specific platform), `ConnectorMetadata`/`ConnectorCapabilities` (declarative coverage and capability discovery — `supports("images")`, `supports("coordinates")`, etc.), `ConnectorConfiguration` (per-instance runtime settings), `ConnectorValidator` (structured, warnings-not-errors-by-default field-completeness checks), and a `ConnectorException` hierarchy (`ConnectorConnectionError`/`ConnectorParsingError`/`ConnectorValidationError`/`ConnectorConfigurationError`).

`ConnectorHealth` was deliberately **not** redefined inside the SDK — it's `src.knowledge.models.ConnectorHealth` from v2.0 Step 4, re-exported and reused via `BaseConnector.health_check()` (which required adding a `platform_id` filter parameter to `knowledge_service.connector_health()`, backward-compatible, defaulting to the old all-platforms behavior). One `ConnectorHealth` concept, not two competing definitions of the same tracked facts.

Both reference connectors (`demo_platform.py`, `demo_platform_two.py`) were rebuilt on `BaseConnector`, removing the real, previously-duplicated fetch->save->parse sequence each independently implemented — every connector now implements exactly four small, genuinely platform-specific hooks: `build_url`, `parse` (raw response -> platform-native records), `normalize` (one record -> one `RawListing`), and `connector_info` (declares `ConnectorMetadata`). `core/agent.py`'s old `_load_connector`/`Connector` ABC were removed outright — nothing needed the `module.CONNECTOR` attribute convention once `ConnectorFactory` existed, and keeping both would have been redundant. `core/agent.py`'s per-platform loop got noticeably simpler too: timing/success/failure now comes from the `ConnectorResult` each connector returns (measured once, inside `BaseConnector.search()`) instead of `core/agent.py` measuring `time.perf_counter()` itself around every connector call, removing a second, redundant measurement.

A small, deliberate, zero-invention design choice: `RawListing` stays exactly where it's always been (`connectors/base.py`), completely untouched — every connector, regardless of source format (HTML today; a future JSON API, GraphQL, RSS, or CSV source), still produces this one shared shape, so nothing downstream of a connector (`analyzers/`, `ranking/`, `search_memory/`, `knowledge/`) needed to change at all. This is the concrete proof that "the Research Agent must never know whether data came from HTML, an API, or a future format" actually holds: not one line changed in any of those packages.

New `docs/18_Connector_SDK.md` — the mission asked for `docs/17_Connector_SDK.md`, but `17` was already taken by Search Memory (v2.0 Step 3); used the next free number and noted the discrepancy rather than silently renumbering anything. Covers architecture, lifecycle, how to build a new connector, best practices, and certification requirements. A new, reusable `tests/connectors/sdk/certification.py` mixin lets any connector's own test file certify SDK compliance (valid metadata, registered, queryable capabilities, returns a `ConnectorResult`, every listing has its required fields) without writing certification logic per connector — both reference connectors' test files now mix this in and pass it.

54 new tests (256 total: 202 existing untouched + 54 new): SDK unit tests for every new class, a template-method test suite using small scripted fake connectors (to test `BaseConnector`'s own orchestration and error-handling independent of any real platform's parsing logic), the certification suite applied to both reference connectors, and registry/factory performance tests proving lookup stays fast with hundreds of registered connectors. Verified against the real dev database, not just tests: ran the actual CLI through the new `ConnectorFactory -> BaseConnector` path end-to-end and confirmed identical results (6 apartments from the same 2 real connectors) plus correct Knowledge Engine observations, exactly matching pre-refactor behavior. A genuine design question resolved deliberately, not left implicit: how does `GeoHistory` learn which provider/method to record, without the calling code (`core/agent.py`) hardcoding a specific provider name and quietly breaking the "provider-independent" guarantee the mission explicitly demands? First draft passed `provider_id="haversine"` as a literal from `_run_geo_engine()` — caught during review as exactly the violation the mission's own "Do not hardcode any map provider" line warns against, even though it was only in a bookkeeping call, not the actual calculation. Fixed by having `record_geo_enrichment()` read `provider_id`/`calculation_method` from the `GeoEnrichment`'s own already-computed `GeoResult`s instead — recorded as `"mixed"` when different travel modes genuinely used different calculation methods (a real, expected case: `"haversine"` for straight-line vs. `"haversine+estimated_speed(5km/h)"` for walking), `"unknown"` only when the enrichment carries no distances at all. `agent.py` never needs to know `"haversine"` exists.
